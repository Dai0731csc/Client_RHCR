# Client Frontend Button Guide

本文面向 `client` 前端页面，说明各个按钮/切换项的用途、前置条件和推荐操作顺序。当前前端主要分为两个页面：

- `https://<host>:8000/`：相机页（Viewfinder）
- `https://<host>:8000/settings`：设置页（Settings）

## 0. 如何访问前端页面

在手机或其他访问设备上打开前端页面之前，需要先找到运行 `client` 的那台电脑的 IP 地址。

### 1. 查询电脑 IP

- Windows：在命令行执行 `ipconfig`
- macOS：在终端执行 `ifconfig`，或在“系统设置 -> 网络”中查看
- Linux：在终端执行 `ip addr` 或 `hostname -I`

通常应查找当前局域网网卡对应的 IPv4 地址，例如 `192.168.1.23`。

### 2. 在浏览器输入访问地址

- 相机页地址：`https://<电脑IP>:8000/`
- 设置页地址：`https://<电脑IP>:8000/settings`

例如，如果电脑 IP 是 `192.168.1.23`，则访问：

- `https://192.168.1.23:8000/`
- `https://192.168.1.23:8000/settings`

### 3. 访问时的注意事项

- 手机或访问设备需要和运行 `client` 的电脑处于同一网络，或能够路由到该电脑
- 浏览器首次访问时，可能会看到 HTTPS 证书提醒，这是因为系统通常使用本地开发证书
- 如果确认访问的是你的目标设备，可以在浏览器中继续访问该页面
- 如果页面无法打开，先检查电脑 IP 是否输入正确、`8000` 端口是否可达、以及 `client` 服务是否已经启动

## 一、相机页按钮说明

页面顶部从左到右共有 6 个主按钮，视频预览区在进入相机标定模式后还会出现 2 个悬浮按钮。

### 1. Open rear camera / Close camera

- 位置：相机页顶部第 1 个按钮
- 作用：打开或关闭后置摄像头
- 使用前提：
  - 页面必须运行在 `HTTPS` 安全上下文中
  - 浏览器需要授予摄像头权限
  - 优先请求后置摄像头；如果设备不支持，会退回到一般的 `environment` 摄像头
- 点击结果：
  - 打开成功后，视频预览开始显示
  - 前端会同步读取当前设备的相机标定信息
  - 如果已有相机内参，后续的连续检测按钮会可用
  - 关闭时会同时停止连续检测、清空当前标定会话，并关闭实时连接
- 推荐用法：进入页面后的第一步先打开相机

### 2. Camera calibration

- 位置：相机页顶部第 2 个按钮
- 作用：进入或退出相机标定模式
- 使用前提：
  - 必须先打开相机
  - 不能与“初始标定”“时间同步”“相机标定上传/采集进行中”同时操作
- 点击结果：
  - 进入模式后，预览区会显示标定进度 `0 / 15`
  - 同时出现两个悬浮按钮：采集棋盘格、上传并计算内参
  - 退出模式后，本轮采集的棋盘格图片会被清空
- 适用场景：首次使用新设备，或设备/镜头参数变化后重新标定

### 3. Time sync

- 位置：相机页顶部第 3 个按钮
- 作用：执行一次浏览器与后端之间的时间同步
- 使用前提：
  - 必须先打开相机
  - 不能和“初始标定”同时执行
- 点击结果：
  - 前端会进行一次时间同步采样
  - 成功后保存最近一次同步结果；再次点击时按钮语义变为“重新同步”
  - 失败时不会阻塞其他按钮，但本次同步结果不会保留
- 推荐用法：正式采集前先执行一次；网络状态变化后可重新执行

### 4. Initial calibration

- 位置：相机页顶部第 4 个按钮
- 作用：采集 AprilTag 标定样本，计算初始参考位姿并发送到后端
- 使用前提：
  - 必须先打开相机
  - 画面中需要稳定看到用于初始标定的标签
  - 当前默认使用的标签 ID 为 `0`
- 点击结果：
  - 前端最多尝试收集 `20` 个有效样本
  - 采样超时、样本不足或未识别到标签时，本次初始标定会失败
  - 成功后会把平均位姿作为 `initial_calibration` 发送给后端
- 推荐用法：在相机位置固定、标签放置稳定后执行

### 5. Start continuous detection / Stop continuous detection

- 位置：相机页顶部第 5 个按钮
- 作用：开始或停止连续 AprilTag 检测
- 使用前提：
  - 必须先打开相机
  - 必须已经存在可用的相机标定内参
  - 如果按钮不可点，通常表示还没有完成相机标定
- 点击结果：
  - 开始后，前端持续检测 AprilTag，并在画面上绘制绿色框和距离标签
  - 检测结果会优先通过 `WebRTC DataChannel` 上报，失败时回退到 `WebSocket`
  - 停止后，检测循环结束，覆盖层清空，并向后端上报 `detection_state=false`
- 推荐用法：完成相机标定后，再开启连续检测

### 6. Open gripper / Close gripper

- 位置：相机页顶部第 6 个按钮
- 作用：切换夹爪开合状态
- 使用前提：
  - 页面可直接点击，不依赖相机是否开启
  - 控制端 server 已 **Start session**，且工具/夹爪链路已启用
  - **本地模式**（`local_udp` / `local_tcp`）：client 通过 **UDP** 发往控制端 `tool_listen`（默认 9002），与 `gripper_service_port` 一致；`local_tcp` 下姿态走 relay，夹爪不走 relay
  - **云端模式**（`cloud_tcp` / `cloud_udp`）：经 cloud relay 的 `gripper_command` 转发
- 点击结果：
  - 当前状态为 `closed` 时，点击发送 `open`
  - 当前状态为 `open` 时，点击发送 `close`
  - 按钮忙碌期间会暂时禁用，避免重复发送
- 注意：前端只负责发送命令；夹爪是否真实动作取决于后端和下游设备链路

## 二、预览区悬浮按钮说明

这两个按钮只会在“相机标定模式”中出现。

### 1. Capture calibration frame

- 位置：视频预览区底部悬浮按钮
- 作用：采集一张棋盘格标定图片
- 使用前提：
  - 已进入相机标定模式
  - 相机画面中清晰可见棋盘格
  - 当前默认需要采集 `15` 张有效图片
- 点击结果：
  - 前端先截取当前视频帧
  - 再调用后端校验当前帧中是否识别到棋盘格
  - 只有校验通过时，计数才会 `+1`
  - 如果未识别到棋盘格，按钮不会报错，但本次不会计入进度
- 当前棋盘格配置：
  - `board_type = chessboard`
  - `rows = 6`
  - `cols = 9`
  - `square_size_mm = 20.0`

### 2. Upload and compute intrinsics

- 位置：视频预览区底部悬浮按钮
- 作用：上传本轮采集的棋盘格图片并计算相机内参
- 使用前提：
  - 已进入相机标定模式
  - 已成功采集满 `15 / 15` 张有效图片
- 点击结果：
  - 前端将图片和棋盘格元数据上传到后端
  - 后端返回标定结果后，前端保存新的相机内参
  - 上传成功后会自动退出相机标定模式
  - 完成后，“连续检测”按钮会变为可用
- 注意：如果上传失败，会保留当前标定模式和采集状态，便于重试

## 三、设置页操作项说明

设置页没有单独的 Save 按钮。所有改动都会自动应用，并立即触发相关出站连接重连。

### 1. ← Camera

- 位置：设置页左上角返回链接
- 作用：返回相机页
- 注意：它只是页面跳转，不会撤销刚刚已经自动应用的设置

### 2. Transport Mode

- 位置：设置页顶部第一组切换项
- 可选项：
  - `local_udp`
  - `local_tcp`
  - `cloud_tcp`
  - `cloud_udp`
- 行为规则：
  - 四项中始终只允许选中一项
  - 点击新选项后会立即提交到 `/api/settings`
  - 提交成功后，后端会按新模式重建 outbound 链路
- 使用建议：
  - `local_tcp` 和 `local_udp` 主要用于本地调试
  - 一般使用时不需要改动 `Transport Mode`
  - 默认保持 `cloud_tcp` 即可
  - 只有在明确进行本地联调或链路排查时，再切换到 `local_tcp` 或 `local_udp`
  - client 与 control server 两端的传输模式必须一致

### 3. Session ID

- 位置：Transport Mode 下方输入框
- 作用：设置当前会话 ID
- 行为规则：
  - 输入后约 `400ms` 自动提交
  - 不需要手动保存
- 使用建议：云端模式下，client 与对端需要使用相同的 `session_id`

### 4. Cloud Host

- 位置：Session ID 下方输入框
- 作用：设置云端 relay 主机地址
- 行为规则：
  - 输入后约 `400ms` 自动提交
  - 对 `cloud_tcp` / `cloud_udp` 模式生效

### 5. Local Topology

- 位置：Cloud Host 下方
- 可选项：
  - `same_machine`
  - `same_lan`
- 行为规则：
  - 两项中始终只允许选中一项
  - 切换后立即自动提交
- 使用建议：
  - client 和 control server 在同一台机器：选 `same_machine`
  - 两端位于同一局域网的不同机器：选 `same_lan`

### 6. Local LAN Host

- 位置：Local Topology 下方输入框
- 作用：填写 client 机器在局域网中的 IP 地址
- 行为规则：
  - 输入后约 `400ms` 自动提交
  - 主要用于 `same_lan` 场景
- 使用建议：填写实际可被 control server 访问到的局域网 IP

## 四、推荐操作顺序

### 首次接入

1. 进入 `Settings`，确认 `Transport Mode`、`Session ID`、`Cloud Host` 或本地网络拓扑
2. 返回相机页，点击 `Open rear camera`
3. 点击 `Camera calibration` 进入相机标定
4. 点击 `Capture calibration frame`，采集满 `15` 张有效棋盘格图片
5. 点击 `Upload and compute intrinsics`
6. 需要时点击 `Time sync`
7. 点击 `Initial calibration`
8. 点击 `Start continuous detection`

### 日常使用

1. 打开相机
2. 如网络或设备状态变化，重新做 `Time sync`
3. 如相机位置参考系需要刷新，重新做 `Initial calibration`
4. 开启 `Start continuous detection`
5. 按需要操作 `Open gripper / Close gripper`

## 五、常见现象说明

- `Start continuous detection` 按钮是灰的：通常表示还没有可用的相机内参，需要先完成相机标定
- 点击采集棋盘格但计数不增加：当前帧没有成功识别到棋盘格，调整角度、距离、光照后重试
- 设置页改完立刻生效：这是设计行为，页面没有 Save 按钮
- 返回相机页后设置仍然保留：设置会写回运行时配置，不会因为页面跳转丢失
- 夹爪按钮点了没动作：前端命令可能已发送，但后端链路或下游设备未正常响应

## 六、操作注意事项

- 完成 `Initial calibration` 后，手机不能立刻移动。应保持手机和相机姿态不变，至少等待 `3` 秒，再点击 `Start continuous detection`。只有进入持续监测后，才可以开始移动手机。
- 一旦 client 与机器人侧链路已经接通，不要在任意位置随意开启或停止 `Start continuous detection`。持续监测一旦停止，应视为本轮控制流程结束。
- 如果中途停止了持续监测，下一次重新开始控制前，必须重新执行 `Time sync` 和 `Initial calibration`。
- 如果中途停止了持续监测，机器人侧也需要先完成复位，再开始下一轮控制，避免沿用上一轮的参考状态。
- 在 `Initial calibration` 和持续监测期间，AprilTag 应保持清晰、完整、尽量正对镜头，避免被手、夹具或其他物体遮挡。
- 相机标定采集棋盘格时，不要连续拍摄几乎完全相同的角度。应覆盖不同距离、位置和轻微倾角，否则内参质量会下降，影响后续检测稳定性。
- 设置页参数会立即生效并触发 outbound 重连。因此在机器人已经准备执行控制时，不要临时修改 `Transport Mode`、`Session ID`、`Cloud Host` 或本地拓扑。
- `local_tcp` 和 `local_udp` 是面向本地调试的模式。若当前是正常远程控制流程，通常不要改动 `Transport Mode`，默认保持 `cloud_tcp`。
- 如果关闭并重新打开相机，建议重新检查当前链路状态，并在正式控制前重新执行 `Time sync` 与 `Initial calibration`，不要直接沿用上一轮的操作节奏。
- `Open gripper / Close gripper` 只代表前端发出了命令，不等于末端执行器一定已完成动作。涉及抓取或释放时，应以机器人侧/夹爪侧的实际反馈为准。
