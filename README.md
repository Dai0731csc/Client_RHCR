# TeleProgram

`TeleProgram` provides an `aiohttp`-based HTTPS/WSS + UDP service. It supports AprilTag detection in the browser, time sync, initial calibration, and camera calibration, and forwards the raw pose stream to the control side over the cloud or a direct local link.

Current goals:

- Real-time AprilTag detection in the browser using the camera
- Prefer uploading detections via `WebRTC DataChannel`, and fall back to `WebSocket` on failure
- Backend acts as `master`, caches state, and broadcasts raw poses to the control-side `slave`

## Install on a new machine

For a fresh setup, use the scripts under `install/`.

macOS / Linux:

```bash
./install/unix/bootstrap_client.sh
```

The install script asks whether to create a `Python 3.11` virtual environment.

- Answer `yes`: install into `client/.venv`
- Answer `no`: skip `.venv` and install directly with `python3.11`

Windows PowerShell:

```powershell
.\install\windows\bootstrap_client.ps1
```

If you want to skip `.venv` and use a specific Python 3.11 interpreter:

```powershell
.\install\windows\bootstrap_client.ps1 -PythonExe "C:\path\to\python.exe" -Venv no
```

Then verify the environment and run the client:

```bash
./install/unix/verify_client.sh
.venv/bin/python main.py
```

```powershell
.\install\windows\verify_client.ps1
.\.venv\Scripts\python.exe main.py
```

If you skip the virtual environment, use:

```bash
./install/unix/verify_client.sh --python python3.11
python3.11 main.py
```

```powershell
.\install\windows\verify_client.ps1
py -3.11 main.py
```

Or use a specific interpreter directly:

```powershell
.\install\windows\verify_client.ps1 -PythonExe "C:\path\to\python.exe"
"C:\path\to\python.exe" main.py
```

`cloud.json` is not generated by the install scripts. Place your cloud config at `config/cloud.json` before the first real run.

On Windows, the certificate generation script depends on `cryptography`, which is included in `install/requirements.txt`.

## Directory structure

- `main.py`: starts the HTTPS + WSS server
- `backend/main.py`: app factory, route registration, startup/shutdown hooks
- `backend/config.py`: static defaults, TLS paths, and accessors that delegate to runtime settings
- `backend/runtime_settings.py`: mutable transport/cloud settings loaded from `cloud.json`
- `backend/wiring/links.py`: builds links and starts the active outbound transport on startup
- `backend/wiring/reconfigure.py`: hot-restarts outbound when settings change
- `backend/links/frontend/`: browser â†” backend (HTTP / WebSocket / WebRTC)
- `backend/links/local/`: LAN UDP pose stream, local `/relay` (`local_tcp` pose only), gripper UDP
- `backend/links/cloud/`: cloud relay (WebSocket or UDP)
- `backend/links/pose_protocol.py`: shared raw-pose JSON protocol constants
- `backend/routers/`: HTTP and WebSocket handlers (including `settings.py`)
- `backend/services/`: ingest, calibration, gripper dispatch, device persistence, settings API
- `backend/models/`: payload and profile shapes
- `backend/utils.py`: timing, timestamps, and ack helpers
- `backend/data/devices/`: device records keyed by IP
- `frontend/camera.html`: camera page (links to Settings)
- `frontend/settings.html`: transport and cloud settings UI
- `frontend/modules/settings.js`: settings page logic (auto-apply, no Save button)
- `frontend/styles/settings.css`: settings page styles
- `frontend/state/`, `frontend/modules/`, `frontend/ui/`: camera workflow
- `frontend/vendor/`: vendored frontend deps and AprilTag wasm
- `config/certificate/local/`: default HTTPS/WSS and local relay TLS files
- `config/cloud.json`: transport mode, cloud host, session id, WebRTC ICE, etc.

## Data flow

1. Browser opens `https://<host>:8000/`
2. The page opens the camera; the browser runs local AprilTag detection
3. Detections go to the client via `WebRTC DataChannel(label=apriltag)` when available
4. If WebRTC fails, the client falls back to `wss://<host>:8000/ws/publish`
5. Initial calibration uses `wss://<host>:8000/ws/calibration/publish`
6. Backend `master` caches `detection_state`, `initial_calibration`, and `apriltag_detections`
7. Backend `master` forwards the stream via `cloud_tcp`, `cloud_udp`, `local_tcp`, or `local_udp`

## Endpoints

### Pages and static assets

- `GET /`: camera page
- `GET /settings`: transport and cloud settings (auto-apply)
- `GET /static/*`: frontend scripts, styles, and vendor assets

### Settings API

- `GET /api/settings`: current runtime snapshot (mode, outbound status, paths)
- `POST /api/settings`: apply changes immediately, persist to `cloud.json`, and reconnect outbound

### WebSocket / WebRTC

- `GET /ws/webrtc`: WebRTC signaling
- `GET /ws/publish`: AprilTag WebSocket uplink
- `GET /ws/calibration/publish`: initial calibration WebSocket uplink
- `GET /relay`: local WebSocket relay (`local_tcp`; control-side server connects as `slave`)
- `GET /api/webrtc/config`: ICE config for `RTCPeerConnection`
- `GET /api/device-profile`: device profile for the request IP
- `GET /api/device-profile/{ip}`: device profile by IP

### Camera calibration and gripper

- `POST /api/camera-calibration/validate`: validate one chessboard frame
- `POST /api/camera-calibration`: upload frames and compute intrinsics
- `POST /api/gripper/command`: gripper open/close (see [Gripper](#gripper) below)

## Configuration (no environment variables)

Settings are read from `config/cloud.json` at startup. While the client is running, use the **Settings** page or `POST /api/settings` to change transport mode and related fields. Changes take effect immediately (outbound reconnects) and are written back to `cloud.json`. You do not need to restart the process.

**Important:** the control-side **server** must use the same `transport_mode` (and matching `session_id` / cloud host when using cloud modes). Mismatched modes will not carry pose traffic.

### WebRTC ICE

In `client/config/cloud.json` under `webrtc` (comma-separated URLs or a JSON array):

```json
"webrtc": {
  "stun_urls": "stun:stun.l.google.com:19302",
  "turn_urls": "turn:your-turn-host:3478?transport=udp",
  "turn_username": "your-username",
  "turn_password": "your-password"
}
```

STUN alone may not traverse all NATs; use TURN for cross-network deployments.

### Local master UDP and gripper

Optional keys in the same file (defaults apply if omitted):

```json
"master_udp_host": "",
"master_udp_port": 9001,
"master_udp_max_packet_bytes": 65507,
"gripper_service_host": "127.0.0.1",
"gripper_service_port": 9002
```

`gripper_service_host` / `gripper_service_port` must match the control server `tool_listen_port` in `server/config/cloud.json` (default **9002**).

### Transport mode

Select exactly one mode with `transport_modes` booleans:

```json
{
  "transport_modes": {
    "local_udp": true,
    "local_tcp": false,
    "cloud_tcp": false,
    "cloud_udp": false
  }
}
```

Local topology (aligned with the control server):

```json
{
  "local_topologies": {
    "same_machine": true,
    "same_lan": false
  },
  "local_lan_host": ""
}
```

- `same_machine=true`: `local_udp` defaults to binding master UDP on `127.0.0.1`
- `same_lan=true`: `local_udp` defaults to `0.0.0.0`; set `local_lan_host` to the client machine IP for the control server

Supported transport modes:

| Mode | Pose stream (master â†’ control) | Gripper (`POST /api/gripper/command`) |
|------|--------------------------------|----------------------------------------|
| `cloud_tcp` | Outbound WSS to `wss://<cloud_host>:<cloud_tcp_port>/relay` | Same cloud relay (`gripper_command`) |
| `cloud_udp` | Outbound UDP to cloud relay (`cloud_host` + `cloud_udp_port`) | Same cloud relay |
| `local_tcp` | Expose `GET /relay`; control connects as `slave` to `wss://<client-ip>:8000/relay` | **Direct UDP** to control `tool_listen` (not relay) |
| `local_udp` | Master UDP on `master_udp_port` (default 9001) | **Direct UDP** to control `tool_listen` |

- `cloud_tcp` / `cloud_udp`: on connect, master runs `clock_sync` with the cloud relay and publishes `skew_cloud_vs_master_ms` via `clock_sync_publish`. The next `detection_state` forwarded to control includes `skew_client_vs_master_ms` (browser) and `skew_cloud_vs_master_ms` (master).
- `local_tcp` / `local_udp`: **gripper never uses** `/relay`; only the pose path uses relay (`local_tcp`) or master UDP (`local_udp`).

Control-side `slave` subscribe:

```json
{"type":"slave_subscribe","protocol":"rhcr-oulu.raw-pose-stream","client_label":"control_AT"}
```

Unsubscribe:

```json
{"type":"slave_unsubscribe","protocol":"rhcr-oulu.raw-pose-stream"}
```

Master responses include `master_stream_ready`, `detection_state`, `initial_calibration`, and `apriltag_detections`.

See [RAW_POSE_PROTOCOL.md](./RAW_POSE_PROTOCOL.md) for the full protocol (timing fields, clock skew, message shapes).

### Clock skew (browser â†” master)

The camera page runs a one-shot clock sync with TeleProgram on load (`frontend/modules/time_sync.js`). When detection starts/stops, `detection_state` includes:

- `skew_client_vs_master_ms` â€” browser wall clock minus master (`T_client - T_master`)
- `clock_sync_rtt_client_ms` â€” optional RTT of that sync

These fields are **not** repeated on every `apriltag_detections` line. In `cloud_*` modes, masterâ†”cloud skew is attached to the same `detection_state` payload before forward (see `backend/links/cloud/clock_skew.py`).

### Gripper

1. Browser calls `POST /api/gripper/command` with `action`: `open` or `close`.
2. **Local modes** (`local_udp`, `local_tcp`): UDP datagram to `gripper_service_host:gripper_service_port` on the control machine.
3. **Cloud modes** (`cloud_tcp`, `cloud_udp`): `gripper_command` is sent through the active cloud relay; the control server executes it on the relay tool ingress path.

Prerequisites on the control server: tool service enabled, robot session started, and matching UDP port if using local modes.

## Notes

- WebRTC is only used for realtime uplink from the browser to this client
- `master` does not compute deltas; it forwards raw poses
- Gripper in **local** modes does not go through `/relay`; align `gripper_service_port` with the server
- Browser â†” client HTTPS uses `config/certificate/local/cert.pem` and `key.pem` by default
- To regenerate local TLS files manually on Windows, run `powershell -ExecutionPolicy Bypass -File .\install\windows\generate_dev_cert.ps1 -PythonCommand python`
